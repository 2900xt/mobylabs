# Stage 1: State Model and Storage

## Goal

Create the durable research state that every agent, TUI view, GUI view, and
artifact writer uses. This is the most important foundation. The product should
not be modeled as chat messages with attachments; it should be modeled as a
research graph with provenance.

## Key Entities

### UserProfile

Captures the human's domain, skills, taste, constraints, preferred methods, and
review preferences.

Fields:

- `id`
- `display_name`
- `fields`
- `expertise`
- `current_interests`
- `preferred_methods`
- `constraints`
- `risk_tolerance`
- `checkpoint_preferences`

### ResearchProgram

Top-level object for a long-lived research direction.

Fields:

- `id`
- `title`
- `description`
- `status`
- `created_at`
- `updated_at`
- `profile_id`
- `workspace_path`

### Paper

Represents an external paper, preprint, technical report, or source document.

Fields:

- `id`
- `title`
- `authors`
- `source`
- `url`
- `doi`
- `arxiv_id`
- `published_at`
- `abstract`
- `summary`
- `embedding_id`
- `ingested_at`

### Claim

A statement extracted from a paper, generated by the agent, or authored by the
human.

Fields:

- `id`
- `program_id`
- `text`
- `claim_type`
- `confidence`
- `source_kind`
- `source_id`
- `created_at`

### OpenQuestion

An unresolved research question.

Fields:

- `id`
- `program_id`
- `text`
- `importance`
- `status`
- `related_claim_ids`

### Hypothesis

A testable proposal with supporting evidence and risks.

Fields:

- `id`
- `program_id`
- `title`
- `statement`
- `why_promising`
- `evidence_ids`
- `risk_summary`
- `feasibility_score`
- `novelty_score`
- `status`

### Experiment

A planned or completed empirical run.

Fields:

- `id`
- `program_id`
- `hypothesis_id`
- `title`
- `plan`
- `datasets`
- `baselines`
- `metrics`
- `status`
- `workspace_ref`

### Result

Structured result from an experiment or proof attempt.

Fields:

- `id`
- `experiment_id`
- `summary`
- `metrics_json`
- `artifact_paths`
- `interpretation`
- `limitations`
- `created_at`

### HumanDecision

A durable record of user judgment.

Fields:

- `id`
- `program_id`
- `checkpoint_id`
- `decision`
- `rationale`
- `created_at`

### ResearchNote

Lab-notebook entry, agent observation, or human note.

Fields:

- `id`
- `program_id`
- `author`
- `note_type`
- `content`
- `linked_entity_ids`
- `created_at`

### DraftSection

A report, paper, memo, or related-work section.

Fields:

- `id`
- `program_id`
- `section_type`
- `title`
- `content`
- `citation_ids`
- `status`
- `updated_at`

## Storage Design

Use SQLite for the local-first MVP.

Recommended layout:

```text
.moby/
  state.sqlite
  artifacts/
    papers/
    notebooks/
    experiments/
    figures/
    reports/
  logs/
```

SQLite is enough for a single-user TUI and early GUI. Postgres can come later
when the app needs concurrent users, hosted projects, shared teams, or remote
workers.

## Implementation Tasks

- Define Pydantic or dataclass domain models.
- Add SQLite migrations.
- Add repository classes for each entity group.
- Add a unit-of-work boundary for multi-entity updates.
- Add import/export to JSON for debugging.
- Add a compact research graph query API.
- Add provenance links between claims, papers, hypotheses, experiments, results,
  and decisions.

## CLI/TUI Surface

Initial commands:

```text
/profile
/program new
/program open
/graph
/notes
/decisions
```

## Complexity

Complexity: L.

Risk level: High. Bad state design will make later agents unreliable and will
turn the product back into chat history. Keep this stage conservative and
well-tested.

## Acceptance Criteria

- A user can create a profile and research program.
- The TUI can open a program and show its current state.
- Papers, claims, hypotheses, experiments, results, notes, and decisions can be
  saved and queried.
- Every generated hypothesis can link back to evidence or explicitly mark
  missing evidence.
- State survives app restarts.
